# Buili

Buili is a construction verification workspace that turns field observations into
review-ready, source-cited RFI, Punch, and Change Evidence packages. It is implemented
against `Buili_Product_Web_Specification_KR.pdf` and includes:

- Next.js PWA frontend for desktop web and phone mode
- FastAPI backend with project, upload, job, issue, RFI, overlay, and report APIs
- SQLite local mode with PostgreSQL-ready SQLAlchemy models
- Ingestion, media, RAG, inference orchestration, and report worker entrypoints
- Model gateway skeleton for KAIST A6000 deployment
- Public-data training pipeline that forces GPU 7 via `CUDA_VISIBLE_DEVICES=7`

The app is intentionally evidence-first: it generates issue candidates with confidence, citations, crops/frames, and model run IDs for human review. Official actions remain human-gated; an AI draft is not an issued construction record.

## Product boundary

Buili is the verification layer before a system of record, not a replacement for
Procore or Autodesk Construction Cloud. Its focused wedge is:

- one traceable Issue Object linking observed condition, exact 2D/3D location,
  current-revision requirement, evidence gaps, review decision, and export version;
- useful spatial context when a project has only 2D drawings or incomplete BIM;
- source-backed RFI/Punch drafting with a mandatory human approval gate; and
- a portable evidence package that can be handed to an existing project workflow.

The lightweight 3D scene is a spatial index. It does not claim fabrication-grade BIM,
as-built reconstruction, automatic clash entitlement, or autonomous construction decisions.

## Quick Start

```bash
cp .env.example .env
# Set BUILI_AUTH_SECRET and BUILI_PILOT_PASSWORD in .env before enabling login.
set -a
source .env
set +a
conda activate cjh_buili
python -m services.api.buili.main
```

In another shell:

```bash
set -a
source .env
set +a
cd apps/web
npm run dev
```

Default URLs:

- Frontend: `http://localhost:3000`
- API: `http://localhost:8000`
- API docs: `http://localhost:8000/docs`

The checked-in example uses SQLite, local object storage, and authentication disabled.
For a local login test, set a random `BUILI_AUTH_SECRET` of at least 32 characters,
set a private `BUILI_PILOT_PASSWORD`, and change `BUILI_AUTH_REQUIRED=true`.
Do not put a real password in source control, screenshots, support tickets, or browser-visible
`NEXT_PUBLIC_*` variables.

### Pilot workspace

On a fresh authenticated database, the optional pilot seed creates one Project Manager
account for Jordan Davis at Northstar Builders and scopes it to the
`Cooper Residence — Electrical Rough-In Verification` project. The workspace contains one authoritative current
electrical drawing, three coherent field-evidence photos, a narrated foreman note with
transcript, and linked review-ready issues. The account email is
`jordan.davis@northstarbuild.example`; its password
is whatever value the operator supplies through `BUILI_PILOT_PASSWORD`.

This is synthetic demonstration evidence, not a real customer record. Set
`BUILI_PILOT_SEED_ENABLED=false` when provisioning an empty organization.

## Quality gates

Run the backend contract and pipeline suites in the project environment:

```bash
conda run -n cjh_buili pytest -q
```

Build the English production web bundle:

```bash
npm run web:build
```

Run the browser acceptance suite after starting the web and API services:

```bash
BUILI_E2E_URL=http://localhost:3000 \
BUILI_E2E_EMAIL="$BUILI_PILOT_EMAIL" \
BUILI_E2E_PASSWORD="$BUILI_PILOT_PASSWORD" \
npx playwright test tests/e2e
```

The specification traceability and known-risk register is maintained in
`docs/product_spec_implementation_matrix.md`.

## Training

The training scripts run inside `cjh_buili` and force GPU 7. They fetch public DocLayNet layout samples when available and train the Buili AI stack baselines:

- PDF drawing/spec RAG reranking
- Plan symbol recognition
- Field media element recognition
- Drawing-field mismatch candidate generation
- Punch list, RFI, and change-order report routing

```bash
npm run train:download
npm run train:full
```

Artifacts and dataset checksums are written to `data/artifacts/buili_ai_stack/`.
Runtime status is exposed at `/v1/training/status`.

## Render Deployment

The repository includes `render.yaml` for a two-service Render deployment:

- `buili-web`: Next.js PWA web app
- `buili-api`: FastAPI core API with a persistent disk for uploaded plans/media/reports
- `buili-postgres`: PostgreSQL database, initialized with the `vector` extension
- `buili-cache`: Render Key Value for queue/cache wiring

Create or update a Render Blueprint from this repo. During Blueprint setup, Render asks
for `BUILI_PILOT_PASSWORD`; enter a unique managed secret of at least 12 characters.
`BUILI_AUTH_SECRET` is generated by Render and neither credential is exposed to the web
service. The web service calls
`/api/*`; the Next.js route handler strips browser-supplied identity headers and proxies the
opaque, revocable HttpOnly session to FastAPI over Render's private network.

To rotate the pilot password, update `BUILI_PILOT_PASSWORD` in Render and redeploy the API.
Startup reconciliation replaces the stored password hash, revokes existing sessions, and
records a `PILOT_PASSWORD_ROTATED` audit event. Unchanged secrets are idempotent.

The Blueprint enables authentication, secure cookies, the single pilot seed, PostgreSQL,
a non-wildcard CORS origin, and persistent API storage. Its health probes are:

- API readiness: `/readyz` (database and evidence storage are checked)
- Web trust path: `/api/readyz` (Next.js proxy to FastAPI readiness)
- API liveness diagnostics: `/healthz`

After deployment, run the non-destructive authenticated smoke check without placing the
password in a command argument:

```bash
export BUILI_SMOKE_URL=https://buili-web.onrender.com
export BUILI_E2E_EMAIL=jordan.davis@northstarbuild.example
export BUILI_E2E_PASSWORD='value stored in Render'
python scripts/smoke_authenticated_deployment.py
unset BUILI_E2E_PASSWORD
```

Optional API environment variables in the Render dashboard are:

- `BUILI_MODEL_GATEWAY_URL` for a KAIST GPU, SGLang, vLLM, or OpenAI-compatible model gateway
- `BUILI_R2_ACCOUNT_ID`, `BUILI_R2_ACCESS_KEY_ID`, `BUILI_R2_SECRET_ACCESS_KEY`, and `BUILI_R2_BUCKET` if object storage is wired in

The API uses the `starter` plan because the evidence store requires a persistent disk.

Manual Render API service disk settings:

- Disk name: `buili-api-storage`
- Mount path: `/var/data`
- Size: `1 GB`
- Environment variable: `BUILI_STORAGE_ROOT=/var/data/buili/storage`

Single-domain Render mode is supported: if the Python API service owns a domain such as
`https://buili.onrender.com`, `GET /` serves the Buili web UI and `/api/*` is accepted as an alias
for the FastAPI routes. This lets uploads and report downloads work with `BUILI_PUBLIC_BASE_URL=/api`.

## Production boundary

The pilot uses password authentication with an opaque, HttpOnly, SameSite session cookie;
the API derives actor, organization, and role server-side. In production,
`BUILI_AUTH_REQUIRED=true` and `BUILI_SECURE_COOKIES=true` are fail-closed settings. Legacy
`X-Buili-Actor` and `X-Buili-Role` headers are development compatibility only and must never
become a browser trust boundary.

Before enterprise rollout, connect the same server-side identity contract to an OIDC/SAML
provider, add managed encrypted object storage, centralized logs/alerts, database and object
version backups, retention/legal-hold controls, and tested recovery objectives. Never expose
the API with SQLite, a wildcard CORS origin, an unset auth secret, or authentication disabled.

Treat source files, evidence hashes, review decisions, audit events, and issued report versions as immutable records. Operational backups must cover both database rows and the corresponding object-storage versions; restoring only one side breaks the evidence chain of custody.
